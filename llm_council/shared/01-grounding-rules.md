# Grounding Rules

## Inputs

A reviewer should prefer these inputs in this order:

1. `review-request.yaml`
2. evidence pack generated by `C-13`
3. relevant changed files or diff snippets
4. targeted repo docs or code snippets if needed

Do not demand the whole repo when the evidence pack and changed files are enough.

## Evidence rules

- Every finding must cite concrete evidence from the provided materials.
- Distinguish facts from inference.
- State uncertainty explicitly when evidence is incomplete.
- Avoid generic advice that is not grounded in the specific change or repo.

## Scope discipline

- Stay inside your persona scope.
- Do not duplicate another persona's primary job unless the risk is severe and clearly overlaps.
- If you notice an issue outside your scope, mention it only as a routed note with the likely owner persona.

## Change discipline

- Treat active runtime paths as primary.
- Treat legacy paths as non-default unless the task explicitly targets them.
- For protocol or auth/session changes, expect cross-surface consequences.

## Output discipline

- Prefer 3 to 5 strongest findings over 15 weak ones.
- Prefer exact fixes over broad opinions.
- Prefer short quoted snippets or path references over long excerpts.
